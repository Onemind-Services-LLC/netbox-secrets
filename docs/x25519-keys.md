# X25519 Key Support

This fork of netbox-secrets adds support for X25519 keys alongside traditional RSA keys. X25519 is a modern elliptic curve Diffie-Hellman (ECDH) algorithm that provides strong security with smaller key sizes and better performance.

## Why X25519?

| Feature | RSA (2048-bit) | X25519 |
|---------|---------------|--------|
| Key size | 2048+ bits | 256 bits |
| Security level | ~112 bits | ~128 bits |
| Performance | Slower | Faster |
| Forward secrecy | No (static keys) | Yes (SealedBox) |

X25519 keys are recommended for new deployments. RSA keys continue to be supported for backwards compatibility.

## How It Works

X25519 encryption uses the **SealedBox** pattern from libsodium:

1. When encrypting the master key for a user:
   - An ephemeral X25519 keypair is generated
   - ECDH key exchange derives a shared secret
   - The shared secret encrypts the master key
   - The ephemeral public key is prepended to the ciphertext

2. Only the user's private key can decrypt the master key

This provides **forward secrecy**: even if the user's private key is compromised later, past communications remain secure (assuming the ephemeral keys were properly discarded).

## Generating X25519 Keys

### Using the Management Command

```bash
# In your NetBox installation
python manage.py generate_x25519_keypair myservice --output-dir ./secrets
```

### Using the Standalone Script

```bash
# Does not require Django/NetBox
python scripts/generate_keypair.py myservice --output-dir ./secrets
```

### Output

Both methods generate two files:

- `myservice.key` - Private key (mode 600, keep secure!)
- `myservice.pub` - Public key (mode 644, upload to NetBox)

### Key Format

X25519 keys use a PEM-like format:

```
-----BEGIN X25519 PUBLIC KEY-----
<base64-encoded 32-byte key>
-----END X25519 PUBLIC KEY-----
```

```
-----BEGIN X25519 PRIVATE KEY-----
<base64-encoded 32-byte key>
-----END X25519 PRIVATE KEY-----
```

## Using X25519 Keys in NetBox

### Creating a UserKey

1. Generate an X25519 keypair
2. Navigate to Secrets > User Keys
3. Click "Add"
4. Paste the contents of your `.pub` file
5. The key type is auto-detected from the header

### API Access

When accessing secrets via the API:

```python
import requests
from nacl.public import PrivateKey, SealedBox
from nacl.encoding import Base64Encoder

# Load your private key
with open('myservice.key', 'r') as f:
    key_pem = f.read()

# Parse the key (strip PEM headers)
key_b64 = key_pem.split('\n')[1]
private_key = PrivateKey(key_b64.encode(), encoder=Base64Encoder)

# Get encrypted master key from API
response = requests.get(
    'https://netbox.example.com/api/plugins/secrets/session-keys/',
    headers={'Authorization': 'Token YOUR_TOKEN'}
)
session_data = response.json()

# Decrypt master key
sealed_box = SealedBox(private_key)
master_key = sealed_box.decrypt(bytes.fromhex(session_data['master_key_cipher']))

# Use master key to decrypt secrets...
```

## Migration from RSA

Existing RSA keys continue to work. To migrate to X25519:

1. Generate a new X25519 keypair
2. Create a new UserKey with the X25519 public key
3. Have an admin activate the new key (copy master key from RSA key)
4. Optionally delete the old RSA UserKey

The system supports mixed key types during migration.

## Security Considerations

1. **Never commit private keys** to version control
2. Store private keys with restrictive permissions (mode 600)
3. Consider using a secrets manager for production deployments
4. X25519 keys are 256 bits but provide ~128-bit security
5. The SealedBox pattern provides forward secrecy
